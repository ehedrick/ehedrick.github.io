Clean Up After Yourself

02 Mar 2014

When I was growing up my dad always made sure I followed one rule whenever we were working on a project around the house: clean up once you are done. This makes logical sense. When everything has a place and it is in that place, then it is easy for yourself or anyone else to pick up on that project without needing to search for that [redacted] 11/64" socket. How come software developers then seem to have such a hard time with this concept?

At some point or another everyone has started on an existing project and run into the brick wall of "How do I get this running?" Now there are tools to help with solving this problem. Java has Maven/Gradle, Ruby has Bundler and Rake, node.js has npm, .NET has nuget, etc. Here's the problem though: these tools tend to suck when you are getting onto a project for the first time. Let's look at an example:

  1. You get some Java project from a GIT repo.
  2. You try to do a mvn clean install and it breaks (shocking) so you yell at it and keep running maven until the build succeeds. This is standard practice after all.
  3. You configure a local Tomcat instance oh wait... You need a data source...
  4. Search for something ending in .sql. Get angry when you notice these require an Oracle database and you only have Postgres.
  5. Spend 3 years setting up express edition.
  6. Notice that there are 20 file to setup and seed the database and you have no clue what order to run them in.
  7. Consult the project guru. Oh there is none and this project hasn't been touched in years? Better have it running before you leave.
  8. Get everything to install with no errors? And a log in screen? Success, or atleast we will assume so.

Now this isn't even a very complex example, just a database and some code that hits it. The question is how do we solve this problem?

First thing is first. Let's tackle the database installation problem that should not be a problem. There are many approaches to fixing this. Some are simple and sometimes work such as numbering the files to mark what order they should run in. Let's take it one step further though: migrations. We will use ActiveRecord as an example. This allows yoou to easily add or remove changes to a database programatically, and lets the developer forget about it. Starting on the project? That's cool. simply run rake db:create and you're done.

Now onto the subject developers won't like: documentation. Yes it's true we don't like it, in fact we despise it. For the most part I say "Bah who needs it". Well we do in this case. Please, please, please make an installation document. Give step by step instructions. Assume you are writting for a monkey at a typewritter computer. Name what database you are using, name what dependencies might be hard to install, give the answers to common problems that might creep up. Yes this is boring, and yes you will spend hours stareing at Word or someother text editor but it will make everyone's lifes so much easier. When you are done writting it remember too: put your tools back. Don't just write it, throw it on a shared drive, and forget about it. Make sure that information stays with the code base. Possibly a project site such as Sharepoint. Or better yet in my opinion: put it in source control. Make it easy to notice. This document is that 11/64" socket that you can't find and will make your code base that much more stable. I don't care how well your code runs because if it isn't stable to install then it isn't a stable code base.
