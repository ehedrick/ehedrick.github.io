Mobile Automation

Jared Sheffer and Ethan Hedrick

Why Bother?

  • No one likes to manually test multiple devices
  • Sanity runs(nightly, on delivery) give quick quality feedback
  • Regression testing is much easier
  • Easily test against multiple OS versions
  • Data comparison is less error prone
  • Easily leverage backend systems (database, services)

Current State of Mobile Automation

Current State of Mobile Automation

  • Messy
  • Multiple tools - no clear winner
    • Most common: Appium and calabash
  • Inconsistent
  • Growing and maturing quickly
  • Similar to web automated testing 5 years ago
  • Bleeding edge

Appium and Calabash similarities

  • No inherent need for cucumber
  • Both can be run from Ruby
  • Both cross platform (iOS and Android)
  • Both can run against simulators and physical devices
  • Both require OSX for iOS testing

Appium Difference

  • GUI tool for element location and interaction
  • Doesn't add anything to the project
  • Isn't tied down to one app
  • No helper tools for cucumber integration
  • Integration with Sauce Labs
  • Reliance on Node.js
  • Does not rely on a specific language

Calabash Differences

  • REPL for element finding and interaction
  • Injects a server into the app
  • Command line tools
  • Integration with Xamarin Test Cloud
  • Has it's own query lanague
  • Version compatibility is a nightmare

Limitations and challenges

  • It can be hard to find elements
  • Tables and scrolling are suboptimal
  • Can only locate elements that are currently visible on screen
  • Execution is problematic
  • Simulator capabilities vs. device capabilities
  • Frameworks can be inconsistent in execution
  • Only one iOS target per machine at any given time
  • Continuous integration can be difficult
